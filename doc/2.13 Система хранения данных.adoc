=== Система хранения данных

==== Назначение хранилищ

|===
|Хранилище |Назначение |Тип данных |Целевые требования

|PostgreSQL
|Источник истины для бизнес-данных чатов
|Профили, чаты, участники, сообщения, метаданные файлов, настройки уведомлений
|ACID, консистентность, транзакции

|KeyDB
|Быстрые ephemeral данные и кэш
|Presence, WebSocket сессии, короткоживущий кэш
|Latency < 5 мс, TTL, высокая RPS

|Kafka
|Шина асинхронных событий
|Domain events (`MessageCreated`, `MessageEdited`)
|Надежная доставка, упорядочивание по ключу, replay

|S3/Object Storage
|Хранение бинарных файлов
|Вложения, аватары
|Дешевое масштабируемое хранение, долговечность

|Opensearch
|Индексация и поиск текстовых сообщений, имен и аккаунтов пользователей
|Текстовые сообщения, профили пользователей
|Быстрая индексация/удаление и поиск всех объектов по совпадению строки
|===

==== PostgreSQL

* Режим - primary + 2 replicas
* Репликация - streaming replication
* Failover - автоматический через оркестратор HA
* Бэкапы:
** full backup - ежедневно
** WAL archiving - непрерывно
** retention backup - не менее 30 дней

==== KeyDB

* Используется для `presence`, `ws_session`
* Все ключи имеют TTL
* Persistence:
** AOF (append only) - включен
** Снапшоты RDB - по расписанию
* При недоступности KeyDB система работает в деградации:
** возможны лишние push
** временно снижается точность online/offline статусов

==== Kafka

* Минимум 3 broker для отказоустойчивости
* replication factor - 3
* min in sync replicas - 2
* Ретеншн:
** `teambridge.chat.message.v1` - 7 дней
** `teambridge.notification.requested.v1` - 3 дня
** `teambridge.audit.v1` - 30 дней
* Partition key для message событий - `chat_id` (сохранение порядка событий внутри чата)

==== S3/Object Storage

* Хранятся только бинарные данные, метаданные - в PostgreSQL
* Доступ к объектам:
** загрузка/чтение - через File Service
* Ретеншн и lifecycle:
** активный класс хранения - первые 90 дней
** архивный класс - по политике компании
* Антивирусная проверка и content type validation выполняются до публикации объекта

==== OpenSearch

* Используется для полнотекстового поиска сообщений и профилей пользователей
* Кластерная конфигурация:
** минимум 3 data nodes для отказоустойчивости
* Индексация:
** Search Service подписывается на Kafka topic `teambridge.chat.message.v1`
** Индексация происходит асинхронно с минимальной задержкой
* Бэкапы:
** snapshot repository в S3
** инкрементальные снапшоты - ежедневно
** retention снапшотов - 30 дней

==== Политика retention по типам данных

|===
|Тип данных |Срок хранения |Основание

|Сообщения и история чатов
|12 месяцев (по умолчанию)
|Бизнес требования и аудит

|Аудит действий администраторов
|12-36 месяцев (зависит от политики компании)
|Комплаенс и расследования

|Presence и WebSocket сессии
|Минуты/часы (TTL)
|Эфемерные операционные данные

|Push delivery status
|30-90 дней
|Диагностика и аналитика доставки

|Индексы сообщений и профилей
|Срок хранения сообщений и истории чатов
|Бизнес требования и аудит
|===

** PostgreSQL Partitioning

|===
|Таблица| Стратегия| Объяснение

| Messages| по временному диапозону на основе created_at (monthly) | Сообщения - данные распределенные во времени. Партиционирование по времени позволяет быстро архивировать или удалять старые сообщения разом. Оптимально для запросов последних сообщений

| Audit Logs| по временному диапозону на основе created_at (monthly) | Логи - данные распределенные во времени. Партиционирование по времени позволяет быстро архивировать или удалять старые данные разом. Оптимально для запросов последних сообщений
