=== Схема защиты и управления доступом

==== Управление доступами с RBAC

*Механизм аутентификации и авторизации*

Использовать связку **OAuth 2.0 + OpenID Connect (OIDC) + JWT** для сценария системы, обслуживающей desktop/web/mobile клиенты.

*Обоснование:*

* *OAuth 2.0 + OIDC* — стандарт для веб- и мобильных клиентов: Authorization Code Flow для интерактивного входа, возможность единого входа (SSO) под корпоративными аккаунтами, делегирование авторизации без передачи пароля приложению.
* *JWT* — самодостаточные токены с подписью (RS256/ES256): проверка на стороне API Gateway по JWKS без обращения к Keycloak на каждый запрос; роли и разрешения в claims ускоряют авторизацию.

*Keycloak* выдаёт и хранит учётные данные, используется для логина и refresh; *API Gateway* валидирует JWT или перенаправляет в keycloack для логина

*Выдача токенов:*

* *Веб-клиент (SPA):* Authorization Code Flow с PKCE. Пользователь перенаправляется на Keycloak; после успешной аутентификации Keycloak возвращает `code` в redirect_uri; Frontend (через API Gateway) обменивает `code` + `code_verifier` на пару access_token (JWT) + refresh_token. Пароль не передаётся во Frontend.
* *Мобильные клиенты (Android/iOS):* тот же Authorization Code Flow с PKCE; токены хранятся в secure storage (Keychain/Keystore).
* *Сервис-сервис (Backend → внешние системы или внутренние вызовы с идентичностью сервиса):* Client Credentials Flow — сервис получает JWT по client_id/client_secret для машинной аутентификации.

Выдача выполняется в Keycloak; в модели это соответствует связям: клиенты → api-gateway, далее api-gateway → keycloack (логин/обмен кода на токены).

*Проверка токенов:*

* *На каждом запросе к API/в момент установления соединения по webScoket:* клиент передаёт `Authorization: Bearer <access_token>`. API Gateway/WebSocket Gateway проверяют JWT: подпись по JWKS (ключи с HTTPS endpoint IAM), проверка `exp`, `iat`, `iss`, `aud`; при необходимости проверка отзыва. После валидации Gateway извлекает `sub`, роли и при необходимости permissions, передаёт контекст (User-ID, роли) в Backend и выполняет маршрутизацию.
* *JWKS:* API Gateway кэширует JWKS от Keycloack и обновляет по TTL или при ошибке проверки подписи; это снижает нагрузку на Keycloack и задержку. В модели это отражено в связи api-gateway → Keycloack «Валидация JWT, JWKS».

* *Проверка `exp`, `iss`, `aud` и извлечение ролей* выполняются **на каждом запросе** из **тела переданного клиентом JWT** (локально, после проверки подписи по ключам из кэшированного JWKS). Срок жизни access token **определяется только** полем `exp` в токене

* *Проверка роли: авторизация* происходит уже в самих backend сервисах. Какждый сервис определеяет доступ к своему АПИ в засимости от Роли. Это обеспечивает защиту АПИ также при межсервисном взаимодействии. 

Таким образом, выдача централизована в Keycloak, проверка — на периметре в API Gateway с опорой на JWT и JWKS, без обязательного вызова Keycloak на каждый запрос.

==== Модель управления доступом: RBAC vs ABAC

*Рекомендация для системы:* **RBAC (Role-Based Access Control)** с проверкой владения ресурсом в бизнес-логике. ABAC (Attribute-Based Access Control) на текущем этапе не вводится.

*RBAC:* решение «разрешить/запретить» принимается на основе *ролей* субъекта (пользователь или сервисная учётная запись) и привязанных к ролям *разрешений* (операция + тип ресурса). Роль определяет, какие действия (READ, WRITE, DELETE, EXECUTE, MANAGE) допустимы. Реализация: роли в JWT, маппинг «роль → разрешения» в каждом backend сервисе.

===== Роли системы и уровни прав

[options="header",cols="2,1,4"]
|===
| Техническое имя (JWT, БД) | Отображаемое имя | Уровень прав и область применения

| ADMIN
| Администратор
| Полный доступ: управление пользователями, ролями, групповыми чатами, уведомлениями, настройками системы; доступ к админ-функциям и мониторингу; MANAGE на всех ресурсах.

| OPERATOR
| Оператор
| Управление всеми групповыми чатами, уведомлениями, просмотр очередей и метрик; READ/WRITE/EXECUTE на запросы и мониторинг; без управления пользователями и ролями.

| USER
| Пользователь
| Создание и управление *собственными* сообщениями, групповыми чатами, профилем; READ/WRITE/EXECUTE только в рамках своих ресурсов (проверка владельца в сервисах).

| SERVICE
| Сервис
| Машинная учётная запись для сервис-сервис вызовов, входящих webhook и фоновых задач. Аутентификация: Client Credentials (OAuth 2.0). Ограниченный набор разрешений: только те endpoint'ы и операции, которые нужны конкретному клиенту (например, приём webhook платёжного провайдера — только POST на /api/v1/webhooks/payments). Не привязана к «своим» данным пользователя; доступ задаётся client_id и scope в keycloak.

| AUDITOR
| Аудитор
| Только чтение логов аудита и событий безопасности (audit_logs, security_events); экспорт отчётов для соответствия; READ на audit; без изменения данных; маскирование PII в экспорте по политике.
|===

*Разрешения (Permissions):*  Маппинг «роль → разрешения по типу ресурса» хранится в конфигурации каждого backend service; при запросе проверяется наличие нужного разрешения у одной из ролей пользователя и при необходимости владелец ресурса.

*Добавление и изменения ролей*

* *Кто может назначать и снимать роли:*

  * Назначение и снятие ролей пользователям выполняют только пользователи с ролью *ADMIN* (Администратор). В Keycloak это реализуется через realm roles и ограничение доступа к админ-консоли

* *Где хранятся роли:*

  * *Keycloak:* роли realm/клиента; при логине роли попадают в JWT (claim `realm_access.roles` или `resource_access.<client>.roles`). Это источник истины для аутентификации и первичной проверки на API Gateway.


*Автоматизация проверки соответствия политик доступа*

  * *Тесты авторизации (unit / integration):* для каждого критичного endpoint'а и роли — тест «пользователь с ролью X получает 200/201 при разрешённой операции» и «пользователь без нужной роли получает 403». Роли USER/VIEWER — дополнительно тесты «доступ к чужому ресурсу (другой user_id) возвращает 403 или 404». Автоматизация: CI (например, в рамках тестов приложения или отдельного набора тестов политик).
  * *Проверка «роль SERVICE»:* для каждого client_id с ролью SERVICE явный перечень разрешённых scope или endpoint'ов; тесты или скрипты проверяют, что сервисный аккаунт не имеет доступа к endpoint'ам, не указанным в перечне.


==== Безопасность передачи данных

*Использование SSL/TLS для защиты данных в сети*

* Клиенты (desktop, web, android, ios) подключаются к api-gateway по HTTPS (TLS 1.3), api-gateway выполняет TLS termination (summary: «распределение трафика, TLS termination»).
* Внутри периметра (API Gateway → Backend) — HTTP/2.
* Клиенты (desktop, web, android, ios) подключаются к ws-gateway по HTTPS и WSS (TLC 1.3), ws-gateway выполняет TLS termination и держит активные websocket подключения.
