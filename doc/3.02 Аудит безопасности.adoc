=== Аудит безопасности


==== Логирование событий безопасности

*Цель:* фиксировать все значимые события, связанные с аутентификацией, авторизацией и доступом к конфиденциальным данным, для расследования инцидентов

===== Какие события логировать

*Аутентификация:*

  * *Успешный вход (LOGIN_SUCCESS):* пользователь успешно аутентифицирован (OAuth2/OIDC, обмен code на токены). Поля: user_id (или subject из JWT), ip_address, user_agent, timestamp, client_id (если применимо). Событие публикуется при успешном логине (API Gateway или отдельный endpoint, вызываемый после обмена токена в Keycloak).
  * *Неуспешный вход (LOGIN_FAILURE):* неверные учётные данные, отозванный или истёкший refresh token, ошибка Keycloak. Поля: ip_address, user_agent, reason (invalid_credentials, token_expired, account_locked и т.д.), timestamp; user_id — только если удалось определить (например, по email в запросе, без логирования самого пароля). Источник: API Gateway
  * *Выход (LOGOUT):* явный logout пользователя, отзыв refresh token. Поля: user_id, session_id, ip_address, timestamp.
  * *Истечение сессии / таймаут неактивности:* при необходимости — событие или учёт в логах (низкий приоритет по объёму).

*Авторизация и доступ к API:*

  * *Отказ в доступе (UNAUTHORIZED_ACCESS / 401, 403):* запрос без валидного токена (401) или с недостаточными правами (403). Поля: request_id, method, path (endpoint), user_id (если токен был передан и валиден), ip_address, user_agent, status_code (401/403), reason (missing_token, invalid_token, insufficient_permissions, forbidden_resource), timestamp. Источник: API Gateway, WS Gateway и любой другой backend сервис — на каждый ответ 401/403 генерировать событие и публиковать в топик audit/security для Audit Service.

*Изменения прав и учётных записей:*

  * Обращения к keecloack Логируется самим Keycloack

*Доступ к секретам (Vault):*

  * Обращения к Vault (чтение секретов, обновление) логируются самим Vault.

==== Ключевые метрики для мониторинга аномального поведения

Метрики предназначены для дашбордов и алертов; сбор — Prometheus/OpenTelemetry из сервисов и из логов

*Аутентификация:*

  * *Количество неуспешных попыток входа в единицу времени* (по всему кластеру и в разбивке по ip_address, user_id/identifier). Алерт: рост выше порога (например, > 50 за 5 мин по одному IP или > 2% от успешных за 10 мин без всплеска легитимного трафика) — возможный brute-force
  * *Соотношение успешных и неуспешных логинов* (rate login_success, rate login_failure). Резкое увеличение доли неуспешных — признак атаки или сбоя Keycloak.
  * *Количество запросов с 401 (Missing/Invalid token)* по endpoint и по IP. Массовый 401 с одного IP — сканирование или неверная настройка клиента.

*Авторизация:*

  * *Количество ответов 403 (Forbidden) в единицу времени* — по user_id, по ip_address, по endpoint (path). Алерт: аномальный рост 403 по одному пользователю или по одному IP (перебор чужих ресурсов, эскалация привилегий).
  * *Доступ к чувствительным endpoint'ам:* rate запросов к /api/v1/admin/* по user_id и по роли. Необычно высокий доступ с роли USER к админ-путям (при корректной проверке прав таких запросов быть не должно — метрика для верификации и расследований).

*Rate limiting и злоупотребления:*

  * *Количество ответов 429 (Too Many Requests)* по user_id, по IP, по endpoint. Алерт при аномальном росте 429 (2.09), в частности на endpoint логина — защита от brute-force.
  * *Топ IP и user_id по количеству отказов (403 + 429 + login_failure)* за скользящее окно (например 1 ч). Использование для ручного ревью и блокировок.


==== Интеграция с Observability и реагированием

  * *Дашборды:* в Grafana выделить дашборд «Безопасность и аудит»: метрики аутентификации/авторизации/rate limit (успех/неудача), 401/403/429 по endpoint и по источнику. При необходимости — ссылки на запросы в логах (request_id) для расследования.
  * *Алерты и runbook:* алерты по перечисленным метрикам направляются в Alertmanager / Grafana Alerting с маршрутизацией в канал «Безопасность»
