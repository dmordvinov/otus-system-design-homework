=== Хранение и управление секретами

Документ описывает выбор инструмента для управления секретами, схему хранения ключей, паролей, токенов и конфигураций, процесс ротации секретов и шифрование конфигурационных данных.

==== Выбор инструмента: HashiCorp Vault

*Решение:* использовать **HashiCorp Vault** как единую систему управления секретами и криптографическими ключами.

  * *Единая точка хранения:* пароли БД, KeyDB, Kafka, API-ключи внешних систем (Push Services), client_secret для Keycloack, ключи шифрования хранятся в Vault; приложения не хранят секреты в коде и по возможности не в переменных окружения на диске.
  * *Аутентификация приложений:* AppRole или Kubernetes Auth Method (Service Acccount) — каждый сервис (или под) получает ограниченный токен с доступом только к своим путям в Vault; принцип наименьших привилегий (NFR-005).
  * *Аудит:* Vault логирует все обращения к секретам (чтение, обновление, создание)
  * *Интеграция со стеком:* Spring Boot (Vault Spring Cloud Vault или Vault Java API), Keycloak, CI/CD — получение секретов при старте приложения или при деплое; при развёртывании в Kubernetes — Vault Agent (sidecar или init) для записи секретов в файлы/переменные окружения.


==== Схема безопасного хранения секретов в Vault

*Структура путей (namespace/path):* использование разных Vault instance, выделение внутри отдельных namespaces для prod и dev + иерархия путей по сервисам и типу секрета снижает риск ошибочного доступа и упрощает политики.

*Какие секреты хранятся и где (логическая схема):*

[options="header",cols="2,2,2,2"]
|===
| Категория
| Пример пути (KV v2 или engine)
| Содержимое
| Кто читает

| Учётные данные БД (рекомендуется dynamic)
| database/creds/chat-service-db, database/creds/file-service-db, … 
| Логин/пароль PostgreSQL (краткосрочные, генерируются Vault)
| Chat Service, File Service, Search Service, Notification Service

| KeyDB
| secret/prod/keydb
| Пароль (и при необходимости TLS client cert)
| Все микросервисы, использующие кэш

| Kafka (Event Bus)
| secret/prod/kafka
| SASL username/password или SSL keystore/truststore
| Микросервисы и мониторинг

| Keycloak
| secret/prod/keycloack
| Admin API client_id/client_secret; client_secret для backend-клиента Keycloak
| API Gateway, Chat Service

| Уведомления (внешняя система)
| secret/prod/notification
| API key
| Notification Service

| TLS (опционально)
| secret/prod/tls/api-gateway
| Сертификат и приватный ключ для edge (или ссылка на выпуск через Vault PKI)
| Api gateway, WS gateway / Ingress
|===

*Движки Vault:*

  * *KV Secrets Engine v2:* статические секреты (пароли KeyDB, Kafka, API keys, webhook secret). Версионирование позволяет откатиться при ошибке; политики Vault ограничивают доступ по path.
  * *Database Secrets Engine:* для PostgreSQL — Vault генерирует краткосрочные учётные записи (TTL, например 1 час) и при необходимости автоматически ротирует их. Сервисы запрашивают новый логин/пароль при старте или по истечении TTL; статические пароли в БД не требуются для приложений.

*Политики доступа (ACL):* для каждого сервиса (AppRole) задаётся минимальный набор путей: только те секреты, которые нужны этому сервису. Пример: Chat Service — только `secret/prod/chat-service`, `database/creds/chat-service-db`, при необходимости `secret/prod/kafka`. Отказ по умолчанию для всех остальных путей.


*Примечание.* Секреты не записываются в конфигурационные файлы в репозитории; приложение получает их из Vault при старте (Spring Cloud Vault, Vault Agent). Ротация мастер-учётной записи БД (которой Vault создаёт динамических пользователей) — по регламенту (напр. 90 дней);

==== Процесс ротации секретов

*Динамические секреты (PostgreSQL через Database engine):*

  * Vault сам генерирует учётные записи с ограниченным TTL (например 1 час). По истечении TTL учётная запись аннулируется в БД (Vault отзывает пользователя).
  * Ротация «встроена»: приложение при следующем запросе к Vault получает новые credentials. Дополнительный процесс ручной ротации не нужен для этих логинов.
  * Ротация статических учётных записей *в самой БД* (например, мастер-пользователь, которым Vault создаёт динамических пользователей): по регламенту (например раз в 90 дней); обновление credentials в Vault Database config и при необходимости перевыпуск динамических токенов.

*Статические секреты (API keys, webhook secret, keyDB, Kafka, Keycloak client_secret):*

  * *Периодичность:* не реже чем раз в 90 дней, по регламенту провайдера или чаще.
  * *Шаги ротации:*
    1. Создание нового секрета (новый API key у push service, новый пароль KeyDB/Kafka и т.д.).
    2. Запись новой версии в Vault (KV v2) по тому же path; при необходимости две версии (старая и новая) для плавного переключения.
    3. Обновление конфигурации потребителя: перезапуск сервисов или обновление через Vault Agent / перечитывание при следующем TTL.
    4. Удаление или деактивация старой версии секрета после перехода.



==== Связь с компонентами решения

  * *Микросервисы:* получают из Vault учётные данные БД (dynamic или static), KeyDB, Kafka, при необходимости — API keys внешних систем (Notification Service — ключи провайдера уведомлений). Конфигурация без секретов; секреты инжектируются при старте.
  * *API Gateway:* из Vault — client_id/client_secret для Keycloak
  * *Keycloak:* пароли пользователей и внутренние секреты Keycloak могут оставаться в Keycloak. Сервисы не хранят пароли пользователей; они хранятся в Keycloak
