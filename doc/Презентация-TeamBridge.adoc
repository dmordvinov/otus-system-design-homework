== Цель и задачи проекта

*Цель* — спроектировать архитектуру корпоративного сервиса для обмена мгновенными сообщениями в режиме реального времени внутри компании.

*Задачи*::

. Зафиксировать функциональные и нефункциональные требования, риски.
. Описать контекст, контейнеры, потоки данных, модель данных, API (OpenAPI), хранение (PostgreSQL, keyDB).
. Спроектировать ИБ системы (OAuth 2.0 / OIDC / JWT, RBAC, Vault).
. Спроектировать мониторинг (метрики, логи, трассировка).
. Задать развёртывание IaC (Terraform, Ansible), контейнеры и Kubernetes, CI/CD.

== Требования

** Realtime-обмен сообщениями (личные и групповые чаты, включая режим каналов)
** Поиск по сообщениям и пользователям
** Файлообмен (до 100 МБ) и push-уведомления
** SSO + LDAP/AD, роли, аудит
** Отказоустойчивость и масштабирование (SLA 99.9%)
* **Целевые метрики** - latency доставки до 300 мс, RPO до 5 мин, RTO до 30 мин
* Защита трафика - TLS 1.3

== Контекст

Система взаимодействует с пользователями (веб, мобильное приложение, desktop), провайдерами уведомлений, keycloak; внутренние сервисы изолированы за API Gateway.

== Архитектурные подходы
* Микросервисная архитектура - упрощает независимое масштабирование и развитие отдельных доменов
* Event Sourcing + событийное взаимодействие через Kafka - надежная асинхронная обработка, replay, слабая связанность
* Разделение API Gateway и WebSocket Gateway - оптимизация под sync API и long-lived realtime-соединения

== API/Realtime:

** REST (внешний контур) - понятный и совместимый интерфейс для клиентов и интеграций
** gRPC (внутренний контур) - более эффективные межсервисные вызовы
** WebSocket Gateway - доставка realtime-событий

== Хранение и кэш

** PostgreSQL (primary + replicas) - транзакционное хранилище бизнес-данных
** KeyDB - presence/сессии и быстрый кэш с TTL
** Kafka - доменные события и асинхронная интеграция
** S3 (MinIO) - хранение вложений
** OpenSearch - полнотекстовый поиск

== Безопасность и управление доступом
- Keycloak + LDAP/AD - SSO и централизованное управление учетными записями
- RBAC с проверкой владельца ресурса в каждом сервисе, 
- Vault для секретов, 
- HTTPS, WSS (TLS 1.3) для внешних коммуникаций
- логи событий безопасности и аудит.

== Observability
*Сбор метрик (VictoriaMetrics), дашборды и алерты (Grafana, Alertmanager)*

* Технические метрики по сервисам (http_5xx_rate, ws_connect_rate, consumer_lag, service_memory_usage, service_cpu_usage, etc..)
* Продуктовые метрики (messages_sent_total, messages_delivered_total, etc...)

*Логи*

* Структурированные JSON логи во всех сервисах
* Обязательные поля - `timestamp`, `service`, `trace_id`, `request_id`, `user_id` (если допустимо), `chat_id`, `event_type`, `status`, `error_code`
* Единая схема ошибок - код, причина, классификация (`transient`/`permanent`)
* Маскирование чувствительных данных - токены, пароли, персональные данные

*Трассировка*

* OpenTelemetry SDK во всех backend сервисах
* Propagation контекста через HTTP и Kafka headers
* Спаны на внешние вызовы: Keycloak, PostgreSQL, Kafka, Redis, S3
